# Story C-WIRE-3: Wire GitHub Actions generator into CLI

## Goal
Add CLI integration for the GitHub Actions generator (`generators/github-actions-generator.js`) which exists but has no CLI entry point.

## Acceptance Criteria
- [ ] AC1: `bmad-swarm init --github` generates the GitHub Actions workflow file in `.github/workflows/`
- [ ] AC2: `bmad-swarm update` does NOT regenerate the GitHub workflow (it is a one-time generation, not a managed file)
- [ ] AC3: If `--github` is not passed, no workflow file is generated (backward compatible)
- [ ] AC4: The generated workflow file matches the output of `generateGitHubWorkflow()` in `generators/github-actions-generator.js`
- [ ] AC5: `bmad-swarm init --help` shows the `--github` flag
- [ ] AC6: Test passes: `npm test -- --grep "github|workflow|init.*github"`

## Dev Notes
- Files to modify:
  - `cli/init.js` -- add `--github` option to the init command; call `generateGitHubWorkflow(projectRoot)` when the flag is present
  - The `generateGitHubWorkflow()` function is already fully implemented in `generators/github-actions-generator.js`. It takes `projectRoot` and writes to `.github/workflows/bmad-validate.yml`.
- Pattern to follow: Look at how `--scan` and `--template` flags are handled in `cli/init.js:24-26`. Add `--github` in the same style.
- The generator already includes a comment `# Generated by bmad-swarm init --github` in the workflow file (line 14), confirming this was the intended integration point.
- This is NOT added to `update` because workflow files are typically committed to git and should not be regenerated/overwritten. They are a one-time generation.
- Test file: `test/cli.test.js` -- add a test for `bmad-swarm init -y --github` that verifies `.github/workflows/bmad-validate.yml` is created.
